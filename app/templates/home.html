How it works

Preparing the data

Every image is converted to greyscale and resized to 64×64. Colour carries no information about whether someone is smiling, and dropping it removes two thirds of the input for free.

The features

Rather than feeding raw pixels to the classifier, each image is turned into a Histogram of Oriented Gradients: the image is divided into 8×8 cells, and each cell records the directions in which brightness changes. A smile is a specific arrangement of curved edges around the mouth, which is exactly what this measures. It is also far more tolerant of lighting than pixel values, because it looks at how brightness changes rather than what it is.

HOG is implemented directly in numpy here rather than imported from scikit-image. It is one well-defined algorithm, and writing it out removed a large compiled dependency with no measurable loss in accuracy.

The model

The 1,764 HOG values pass through standardisation, then PCA keeping 95% of the variance, then an RBF-kernel support vector machine. With around 1,200 training images and more feature dimensions than samples, an SVM is the right classical choice: it optimises the margin between the two classes rather than trying to model the whole space. A neural network would need far more data, a GPU, and a much larger container to solve a problem that fits in two seconds of CPU time.

Preprocessing and classifier are wrapped in a single scikit-learn Pipeline and pickled together, so inference can never drift from training: there is one artifact, and it contains every transformation applied at fit time.

Honest reporting

The dataset is close to balanced, 600 smiling against 603 not smiling, so accuracy is a meaningful number on its own. F1 for the smiling class is reported next to it as a check: if the two ever diverge, the model has started favouring one class and the accuracy figure has stopped meaning what it appears to.
